React组件的生命周期

就像人有生老病死，react中的组件也有自己的生命周期，每个组件在被创造出来之后，都可能经历更新和删除的过程，就如同鲜活的个体。

闲话少说，我们进入正题👇👇

React严格定义了组件的生命周期，组件创建之后会经历以三个过程：

  • 装载过程（Mount）：将组件第一次在DOM树中进行渲染时
  • 更新过程（Update）：组件被再次渲染时
  • 卸载过程（Unmount）：将组件从DOM树中删除时

react库会不同的时期调用react组件的一些成员函数来实现生命周期的更替，这些函数也被称为生命周期函数。在具体化过程之前，我们可以看看下面这张图：

1.装载过程（Mount）

在组件第一次被渲染的时候，调用的是下面这些函数：

  • constructor
  • getInitialState
  • getDefaultProps
  • componentWillMount
  • render
  • componentDidMount

constructor是ES6中每个类的构造函数，一个react组件如果需要构造函数，往往是需要为了两个目的：

1.初始化state
2.绑定成员函数的this环境

getInitialState这个函数的返回值会用来初始化的组件的this.state，getDefaultProps这个函数的返回值可以作为props的初始值，但是这两个函数值在React.createClass方法创造的组件类才会用到。换句话说，这两个方法在ES6定义的React组件中根本不会用到。

毋庸置疑，render函数是整个React组件中最重要的函数，render函数虽然是渲染函数，但是它却不做实际的渲染动作，而是返回一个JSX，最终还是由React来完成渲染的工作。值得注意的是render是一个纯函数，若在render函数中调用this.setState毫无疑问是错误的。

从上面的Mount过程中，我们可以清楚看到componentWillMount和componentDidMount分居在render两侧，顾名思义，这两个函数将在组件挂载前后分别起作用。componentWillMount发生在组件将要装载的时候，也就是说，我们完全可以把在componentWillMount中要做的事情放到constructor中去做。

但是componentDidMount的作用就大了，render函数调用之后并不会立刻调用componentDidMount，而是等待render函数被触发且组件挂载到DOM树之后，才会被调用。componentDidMount将作为挂载过程的收尾。

2.更新过程（Update）

装载完成之后，展现在用户面前的将是React组件的初印象，随着用户的操作，组件的展现形式必然要发生变化，这时候将会引发组件的更新过程。

更新过程将会调用下面这些函数：

  • componentWillReceiveProps
  • shouldComponentUpdate
  • componentWillUpdate
  • render
  • componentDidUpdate

当props发生变化时执行，初始化render时不执行，在componentWillReceiveProps里面，可根据属性的变化，通过调用this.setState()来更新组件状态，旧的属性还是可以通过this.props来获取,这里调用更新状态是安全的，并不会触发额外的render。

shouldComponentUpdate返回一个布尔值。在组件接收到新的props或者state时被调用。如果返回了true，则告诉React库这个组件在这次更新过程中需要重新渲染。

如果组件的shouldComponentUpdate函数返回了true，React接下来就会依次调用对应组件的componentWillUpdate，render和componentDidUpdate。componentWillUpdate在组件接收到新的props或者state但还没有被渲染时被调用，在初始化时不会被调用。componentDidUpdate在组件完成更新后立即调用，在初始化时不会被调用。

3.卸载过程（Unmount）

React组件的卸载过程只涉及到一个componentWillUnmount函数，当组件要从DOM树上删除时，该函数被调用做一些清理性的工作。卸载十分干净利落，一个函数就搞定！
